# Appendix. AI 전문가 질의 대응 Q&A

본 문서는 AI/백엔드/데이터 전문가가 발표에서 물어볼 가능성이 높은 질문을 기준으로 정리한 기술 부록입니다.  
기능 소개보다 **모델 선정 근거, RAG 설계, 평가 방법, 보안, 운영 가능성, 한계와 개선 방향**에 초점을 둡니다.

---

## 1. 핵심 아키텍처 요약

본 프로젝트의 AI 구조는 하나의 LLM에 모든 작업을 맡기는 방식이 아니라, 입력 데이터의 성격에 따라 전문 모델을 분리한 파이프라인입니다.

| 단계 | 모델/기술 | 목적 |
| --- | --- | --- |
| STT | WhisperX | 회의 음성을 발화 시간과 segment가 포함된 transcript로 변환 |
| OCR | PaddleOCR-VL, Qwen VL | PDF/이미지 문서에서 텍스트와 시각 문맥 추출 |
| Parsing | MinerU | 문서의 제목, 본문, 표, 페이지 구조 분석 |
| Embedding | 한국어 Sentence Embedding | 문서/회의록/질문을 의미 벡터로 변환 |
| Vector DB | Qdrant | 프로젝트별 문서와 회의록 chunk 검색 |
| Generation | vLLM 기반 Instruction LLM | 회의록, 안건, 준비자료, 챗봇 답변 생성 |

```text
회의 음성 / 문서 업로드
        ↓
STT / OCR / Parsing
        ↓
Parent-Child chunk 생성
        ↓
Embedding
        ↓
Qdrant project collection 저장
        ↓
RAG retrieval
        ↓
LLM generation
        ↓
회의록, 안건, 준비자료, 챗봇 답변
```

## 2. 전문가 예상 질문과 답변

## Q1. 왜 하나의 LLM이 아니라 여러 모델을 조합했나요?

회의 자동화는 음성 인식, 문서 인식, 검색, 생성이 결합된 문제입니다. 이 작업들은 입력 형태와 최적화 기준이 다르기 때문에 하나의 LLM에 모두 맡기는 구조는 정확도, 비용, 운영 측면에서 불리합니다.

| 작업 | 적합한 모델 유형 | 이유 |
| --- | --- | --- |
| 회의 음성 처리 | STT 모델 | 긴 음성, 발화 시간, segment 처리가 필요 |
| 문서 이미지 처리 | OCR/VLM | PDF, 표, 이미지 기반 텍스트 추출 필요 |
| 내부 지식 검색 | Embedding + Vector DB | 최신 사내 문서를 빠르게 검색해야 함 |
| 최종 답변 생성 | Instruction LLM | 검색된 근거를 요약하고 구조화해야 함 |

핵심은 모델을 기능별로 분리해 각 단계의 품질을 독립적으로 개선할 수 있게 만든 점입니다.

---

## Q2. 이 구조의 가장 큰 기술적 차별점은 무엇인가요?

회의록 생성에서 끝나는 것이 아니라, 생성된 회의록과 업로드 문서를 다시 벡터 DB에 적재해 이후 회의 준비자료와 챗봇 답변에 재사용하는 순환형 지식 구조입니다.

```text
회의 발생
  ↓
회의록 생성
  ↓
회의록을 Qdrant에 적재
  ↓
다음 회의 준비자료와 챗봇 답변의 근거로 재사용
```

따라서 시간이 지날수록 프로젝트별 지식 베이스가 축적되는 구조입니다.

---

## Q3. WhisperX를 선택한 이유는 무엇인가요?

WhisperX는 Whisper 계열의 강한 음성 인식 성능을 활용하면서 alignment와 diarization 확장이 가능하다는 점 때문에 회의 데이터에 적합합니다.

| 후보 | 장점 | 한계 |
| --- | --- | --- |
| Whisper | 범용 STT 품질이 높음 | 발화 정렬과 화자 분리는 별도 구성 필요 |
| faster-whisper | 추론 속도와 메모리 효율 우수 | 회의 구조화 정보는 별도 구현 필요 |
| WhisperX | alignment/diarization 확장 가능 | diarization은 추가 설정과 권한 필요 가능 |
| Clova Speech | 한국어 서비스 친화적 | 외부 API 의존과 비용 고려 필요 |
| Google Speech-to-Text | 안정적인 관리형 API | 외부 전송과 비용 이슈 |

회의록 생성에서는 단순 텍스트보다 발화 구간과 speaker segment가 중요하므로 WhisperX가 더 적합합니다.

## Q4. STT 품질은 어떻게 평가할 수 있나요?

STT는 단순히 transcript가 생성되는지보다 회의록 생성에 필요한 정보가 유지되는지를 봐야 합니다.

| 평가 항목 | 지표/방법 |
| --- | --- |
| 음성 인식 정확도 | WER, CER |
| 발화 누락 | 주요 발화 누락률 |
| 시간 정렬 | segment start/end 오차 |
| 화자 분리 | speaker diarization error rate |
| downstream 품질 | 회의록의 결정사항, todo 추출 정확도 |

발표에서는 “STT 단독 정확도뿐 아니라 회의록 생성 결과의 품질까지 함께 봐야 한다”고 답하는 것이 좋습니다.

## Q5. PaddleOCR-VL을 기본 OCR로 둔 이유는 무엇인가요?

이 프로젝트의 문서 입력은 단순 이미지보다 PDF, 회의자료, 표, 내부 정책 문서에 가깝습니다. PaddleOCR-VL은 문서 이미지와 레이아웃이 포함된 자료를 처리하는 데 적합합니다.

| 후보 | 장점 | 한계 |
| --- | --- | --- |
| Tesseract | 가볍고 오픈소스 | 복잡한 한국어 문서와 표 인식에 한계 |
| EasyOCR | 다국어 지원과 쉬운 사용 | 긴 PDF/레이아웃 처리 한계 |
| PaddleOCR-VL | 문서 OCR과 레이아웃 처리에 강점 | serving과 GPU 자원 관리 필요 |
| Qwen VL | 시각 문맥 이해 가능 | 모든 OCR 요청에 쓰기에는 추론 부담 큼 |
| Google Vision | 관리형 API 품질 우수 | 외부 전송과 비용 이슈 |

---

## Q6. Qwen VL은 왜 보조 경로인가요?

Qwen VL은 OCR 전용 모델이라기보다 이미지와 텍스트를 함께 이해하는 멀티모달 모델입니다. 따라서 복잡한 이미지 해석이나 문서 시각 질의에는 장점이 있지만, 모든 문서 OCR에 사용하면 비용과 latency가 증가합니다.

기본 OCR은 PaddleOCR-VL로 처리하고, 시각적 문맥 해석이 필요한 경우 Qwen VL을 대체 경로로 두는 것이 운영상 합리적입니다.

## Q7. MinerU를 문서 파싱에 사용한 이유는 무엇인가요?

RAG에서 중요한 것은 텍스트 추출량이 아니라 검색 가능한 구조를 만드는 것입니다. MinerU는 PDF의 제목, 본문, 표, 페이지 구조를 분석해 RAG용 chunk 생성에 필요한 문서 구조를 제공합니다.

| 후보 | 장점 | 한계 |
| --- | --- | --- |
| PyMuPDF | 빠른 텍스트 추출 | 레이아웃/표 구조 보존 한계 |
| pdfplumber | 표와 좌표 기반 추출 가능 | 파이프라인 직접 설계 필요 |
| unstructured | 다양한 파일 처리 | 결과 일관성 검증 필요 |
| LlamaParse | RAG 친화적 파싱 | 외부 서비스 의존 가능성 |
| MinerU | PDF 구조 기반 파싱에 적합 | 설치와 실행 환경이 무거움 |

## Q8. Chunk size와 overlap은 어떻게 정했나요?

현재 기준은 `CHUNK_SIZE=650`, `CHUNK_OVERLAP=100`입니다. 이는 검색 정확도와 문맥 보존 사이의 균형을 위한 설정입니다.

| 값 | 의도 |
| --- | --- |
| chunk size 650 | 너무 짧아 문맥이 끊기지 않도록 하면서 검색 단위를 적절히 제한 |
| overlap 100 | chunk 경계에서 중요한 문장이 잘리는 문제 완화 |

전문가 질문에는 “최종적으로는 retrieval 평가셋을 만들어 top-k hit rate, MRR, 답변 근거 일치율 기준으로 튜닝해야 한다”고 답하는 것이 안전합니다.

## Q9. Parent-Child chunk 구조를 사용한 이유는 무엇인가요?

작은 chunk는 검색에는 유리하지만 답변 생성 시 문맥이 부족합니다. 반대로 큰 chunk는 문맥은 충분하지만 검색 정확도가 떨어질 수 있습니다.

Parent-Child 구조는 이 trade-off를 줄이기 위한 설계입니다.

| 구분 | 역할 |
| --- | --- |
| Child chunk | 벡터 검색 대상 |
| Parent context | 문서명, 페이지, 제목, 조항, 표 등 상위 문맥 |

검색은 child로 정밀하게 수행하고, LLM에는 parent context를 함께 제공해 답변 품질과 근거 추적성을 높입니다.

## Q10. Parent-Child chunk가 실제로 좋아졌는지 어떻게 검증하나요?

flat chunk와 Parent-Child chunk를 같은 질문셋으로 비교해야 합니다.

| 비교 항목 | 측정 방식 |
| --- | --- |
| 검색 적중률 | 정답 문서가 top-k 안에 포함되는지 |
| 근거 정확도 | 답변이 실제 문서 근거와 일치하는지 |
| 답변 완성도 | 문맥이 끊겨 답변이 불완전해지는 비율 |
| citation 품질 | 문서명, 페이지, 조항 추적 가능성 |

발표 답변은 “Parent-Child 구조는 이론적 장점이 있고, 정량 검증은 flat chunk baseline과의 retrieval 평가로 진행할 수 있다”가 적절합니다.

## Q11. 왜 한국어 임베딩 모델을 사용했나요?

검색 대상이 한국어 회의록과 내부 문서이기 때문입니다. 질문과 문서가 모두 한국어인 경우, 한국어 문장 의미를 잘 표현하는 임베딩 모델이 필요합니다.

| 후보 | 장점 | 한계 |
| --- | --- | --- |
| Ko-SRoBERTa 계열 | 한국어 문장 유사도에 적합 | 최신 대형 모델 대비 성능 한계 가능 |
| KoSimCSE | 한국어 semantic similarity에 강점 | 모델별 편차 검증 필요 |
| multilingual-e5 | 다국어 검색에 적합 | 한국어 도메인 문서 평가 필요 |
| BGE-M3 | 다국어, 긴 문서, hybrid retrieval에 강점 | 모델 크기와 운영 비용 증가 |
| OpenAI embedding | 품질과 안정성 우수 | 외부 API와 데이터 전송 고려 필요 |

---

## Q12. 임베딩 모델을 바꾸면 기존 벡터는 어떻게 되나요?

임베딩 모델이 바뀌면 기존 벡터와 새 벡터의 공간이 달라집니다. 따라서 기존 Qdrant collection을 그대로 섞어 쓰면 안 됩니다.

대응 방식은 다음과 같습니다.

| 방식 | 설명 |
| --- | --- |
| collection versioning | 모델명 또는 embedding version을 collection 이름/metadata에 포함 |
| full re-indexing | 기존 문서를 새 임베딩 모델로 다시 벡터화 |
| A/B evaluation | 새 모델 collection과 기존 collection을 같은 질문셋으로 비교 |

즉, 임베딩 모델 교체는 단순 설정 변경이 아니라 re-index 전략이 필요합니다.

## Q13. Qdrant를 사용한 이유는 무엇인가요?

Qdrant는 벡터 검색과 payload metadata 관리가 모두 필요한 RAG 시스템에 적합합니다.

| 후보 | 장점 | 한계 |
| --- | --- | --- |
| FAISS | 빠른 로컬 벡터 검색 | metadata, API, 운영 기능 직접 구현 필요 |
| Chroma | 개발 초기 사용이 쉬움 | 운영 규모와 관리 기능 검토 필요 |
| Elasticsearch | 키워드 검색과 운영 경험 풍부 | 벡터 중심 RAG에는 튜닝 필요 |
| Pinecone | 관리형 벡터 DB | 외부 서비스 비용과 데이터 위치 고려 |
| Qdrant | 벡터 검색, metadata filter, collection 관리에 적합 | 별도 서버 운영 필요 |

프로젝트별 collection 분리를 통해 문서 접근 범위를 제어하기 쉽다는 점도 중요합니다.

---

## Q14. 프로젝트별 collection만으로 권한 분리가 충분한가요?

collection 분리는 기본적인 데이터 격리 수단이지만, 그 자체만으로 완전한 권한 관리라고 보기는 어렵습니다.

권한 제어는 다음 레이어를 함께 적용해야 합니다.

| 레이어 | 역할 |
| --- | --- |
| Django 인증/인가 | 사용자가 접근 가능한 project_id 검증 |
| AI 요청 단계 | 백엔드가 검증된 project_id만 AI 서버에 전달 |
| Qdrant collection | 프로젝트별 데이터 물리/논리 분리 |
| payload metadata | meeting_id, document_id, owner, scope 등 추가 필터 |
| audit log | 누가 어떤 프로젝트 문서를 조회했는지 추적 |

따라서 답변은 “collection 분리는 권한 관리의 한 요소이고, 실제 권한 검증은 백엔드 인증/인가와 함께 구성한다”가 정확합니다.

## Q15. 왜 Private RAG가 필요한가요?

사내 배포를 전제로 하면 회의록, 녹음 파일, 내부 문서, embedding vector 모두 민감 정보가 될 수 있습니다.

Private RAG는 문서 원문, 벡터 DB, 검색 로그, 생성 로그를 통제 가능한 인프라 안에서 관리하는 구조입니다.

| 관점 | 외부 API 중심 | Private RAG |
| --- | --- | --- |
| 데이터 위치 | 외부 전송 가능성 | 내부망 또는 통제 서버 |
| 접근 제어 | 외부 서비스 정책 의존 | 사내 권한 체계와 연계 |
| 감사 | 제한적일 수 있음 | 문서, 사용자, 프로젝트 단위 추적 가능 |
| 보관 정책 | 외부 정책 확인 필요 | 사내 정책에 맞춰 직접 관리 |
| 모델 통제 | 제공 모델 범위 내 선택 | 자체 모델 교체 가능 |

핵심은 비용 절감보다 데이터 통제와 보안 요구사항 대응입니다.

---

## Q16. Embedding vector도 민감 정보인가요?

네. embedding vector는 원문을 그대로 저장하지 않더라도, 원문 의미를 반영한 파생 데이터입니다. 따라서 민감 정보로 취급하는 것이 안전합니다.

관리 방안은 다음과 같습니다.

| 항목 | 대응 |
| --- | --- |
| 저장 위치 | 내부망 또는 통제 가능한 DB |
| 접근 권한 | 프로젝트/사용자 권한 기반 제한 |
| 삭제 정책 | 원문 삭제 시 관련 vector도 삭제 |
| 로그 관리 | query, retrieved context, generated answer의 민감정보 마스킹 |
| 백업 | 암호화와 보관 기간 정책 적용 |

## Q17. LLM hallucination은 어떻게 줄이나요?

이 프로젝트는 LLM이 단독으로 답변하지 않고, 가능한 경우 Qdrant에서 검색한 근거를 context로 받아 답변하도록 설계합니다.

| 방법 | 효과 |
| --- | --- |
| RAG retrieval | 내부 문서/회의록 근거 제공 |
| Parent-Child chunk | 검색 정확도와 문맥 보존을 동시에 확보 |
| 프로젝트별 collection | 다른 프로젝트 문서가 섞이는 문제 방지 |
| prompt 분리 | 회의록, 안건, 준비자료, 챗봇 목적별 지시 최적화 |
| no-evidence policy | 근거 부족 시 무리한 답변을 줄임 |

전문가 질문에는 “환각을 완전히 제거할 수는 없고, 검색 근거 기반 생성과 평가셋으로 관리한다”고 답하는 것이 좋습니다.

---

## Q18. RAG 대신 fine-tuning을 하지 않은 이유는 무엇인가요?

내부 문서와 회의록은 계속 변합니다. fine-tuning은 모델의 행동 양식이나 형식을 학습시키는 데 유리하지만, 자주 바뀌는 최신 지식을 반영하기에는 비효율적입니다.

| 기준 | RAG | Fine-tuning |
| --- | --- | --- |
| 최신 문서 반영 | 문서 ingest 후 즉시 가능 | 재학습 필요 |
| 근거 추적 | 가능 | 어려움 |
| 비용 | 검색/저장 비용 중심 | 학습 비용 발생 |
| 적합한 목적 | 내부 지식 기반 답변 | 출력 형식, 말투, 특정 태스크 패턴 |

따라서 현재 단계에서는 RAG가 더 적합하고, fine-tuning은 추후 회의록 출력 형식 고도화에 적용할 수 있습니다.

---

## Q19. vLLM을 사용한 이유는 무엇인가요?

vLLM은 LLM을 OpenAI-compatible API로 serving하기 좋고, GPU 환경에서 추론 효율을 높일 수 있습니다.

| 방식 | 장점 | 한계 |
| --- | --- | --- |
| Transformers 직접 실행 | 자유도 높음 | 동시성, serving, 메모리 관리 부담 |
| llama.cpp | GGUF 모델 운영 가능 | 고성능 GPU serving에는 한계 가능 |
| Ollama | 로컬 테스트 편리 | 운영형 동시 처리에는 추가 검토 필요 |
| OpenAI API | 품질과 안정성 우수 | 사내 데이터 외부 전송 고려 |
| vLLM | GPU serving과 API 호환성에 강점 | GPU 환경 구성 필요 |

---

## Q20. INT4/AWQ 모델을 고려한 이유는 무엇인가요?

회의록 생성과 RAG 답변은 입력 context가 길고, GPU 메모리 사용량이 큽니다. INT4/AWQ 계열 모델은 메모리 사용량을 줄여 제한된 GPU 환경에서 serving 가능성을 높입니다.

| 모델 형태 | 장점 | 한계 |
| --- | --- | --- |
| FP16 대형 모델 | 품질 우수 가능성 | GPU 메모리 요구량 큼 |
| INT4/AWQ | 메모리 효율과 운영 가능성 | 일부 품질 저하 가능성 |
| 소형 모델 | 빠르고 저렴 | 긴 문맥과 복잡한 추론 품질 한계 |

전문가에게는 “품질과 운영 가능성의 trade-off로 양자화 모델을 고려했다”고 답하면 됩니다.

---

## Q21. Top-k 값은 어떻게 정했나요?

README 기준 챗봇 검색은 `CHAT_RAG_TOP_K=5`입니다. top-k는 검색 recall과 LLM context 오염 사이의 균형입니다.

| top-k가 너무 작을 때 | top-k가 너무 클 때 |
| --- | --- |
| 필요한 근거를 놓칠 수 있음 | 관련 없는 문서가 context에 섞임 |
| 답변 근거 부족 | LLM 입력 길이 증가 |
| 특정 표현 불일치에 취약 | 답변 집중도 저하 |

최종 값은 질문셋 기반으로 top-k hit rate, MRR, 답변 근거 일치율을 측정해 조정하는 것이 바람직합니다.

---

## Q22. Reranker를 기본 적용하지 않은 이유는 무엇인가요?

Reranker는 검색 결과 품질을 높일 수 있지만 latency와 비용을 증가시킵니다.

| 방식 | 장점 | 한계 |
| --- | --- | --- |
| vector search only | 빠르고 단순 | 상위 결과 품질이 부족할 수 있음 |
| reranker 추가 | 최종 context 정밀도 향상 | 비용과 응답 시간 증가 |
| hybrid search | 키워드와 의미 검색 보완 | 인덱스와 튜닝 복잡도 증가 |

따라서 기본은 vector search로 두고, 정확도가 중요한 기능에 선택적으로 reranker를 적용하는 방향이 합리적입니다.

## Q23. 검색 실패 시 어떻게 처리하나요?

검색 결과가 없거나 similarity가 낮은 경우 LLM이 무리하게 답하지 않도록 해야 합니다.

권장 정책은 다음과 같습니다.

| 상황 | 처리 방식 |
| --- | --- |
| 관련 문서 없음 | 근거를 찾지 못했다고 응답 |
| 낮은 유사도 | 일반 답변보다 추가 정보 요청 |
| 여러 문서 충돌 | 출처별 차이를 명시 |
| 권한 없는 문서 | 검색 대상에서 제외 |

발표에서는 “검색 실패는 정상 케이스로 보고, no-evidence 응답 정책을 둔다”고 답하는 것이 좋습니다.

## Q24. 평가셋은 어떻게 구성해야 하나요?

전문가 발표에서는 “좋아 보인다”보다 재현 가능한 평가셋이 중요합니다.

| 평가셋 | 구성 |
| --- | --- |
| STT 평가셋 | 회의 녹음 일부와 사람이 작성한 정답 transcript |
| OCR 평가셋 | PDF/이미지와 사람이 검수한 텍스트 |
| Retrieval 평가셋 | 질문, 정답 문서, 정답 chunk |
| Generation 평가셋 | transcript와 사람이 만든 회의록 |
| RAG QA 평가셋 | 질문, 근거 문서, 기대 답변 |

| 영역 | 지표 |

| 검색 | top-k hit rate, MRR, recall@k |

## Q26. 회의록 생성 품질은 어떻게 평가하나요?

회의록은 요약문이 아니라 업무 산출물이므로 일반 요약 평가만으로는 부족합니다.

| 평가 항목 | 설명 |
| --- | --- |
| 결정사항 추출 | 회의에서 확정된 내용이 빠지지 않았는지 |
| Todo 추출 | 담당자, 작업, 기한이 정확한지 |
| 발화 왜곡 여부 | 실제 발언과 다른 내용이 생성되지 않았는지 |
| 중복/누락 | 주요 안건이 빠지거나 중복되지 않았는지 |
| 업무 사용성 | 사용자가 바로 등록/수정 가능한 형태인지 |

## Q27. 긴 회의 녹음이나 큰 PDF는 어떻게 처리하나요?

긴 작업은 동기 API보다 job API로 처리합니다.

```text
1. 작업 생성
2. job_id 반환
3. 상태 polling
4. 완료 후 결과 조회
```

| 문제 | 대응 |
| --- | --- |
| HTTP timeout | job API 사용 |
| GPU 작업 충돌 | worker 수 제한, lock 사용 |
| 큰 파일 처리 | 임시 저장 후 단계별 처리 |
| 실패 추적 | job 상태와 error 기록 |

## Q28. 현재 job queue는 운영 환경에 충분한가요?

현재 구조가 메모리 기반 job 관리라면 단일 서버 테스트에는 적합하지만, 운영 환경에서는 한계가 있습니다.

운영 고도화 방향은 다음과 같습니다.

| 한계 | 개선 |
| --- | --- |
| 서버 재시작 시 job 상태 유실 | DB 기반 job 상태 저장 |
| 멀티 인스턴스 확장 어려움 | Redis Queue, Celery, Dramatiq 등 사용 |
| 실패 재시도 제한 | retry/backoff 정책 추가 |
| 작업 우선순위 없음 | queue priority 적용 |

전문가 질문에는 “현재는 기능 검증 중심이고, 운영 배포 시 queue와 job state를 영속화해야 한다”고 답하는 것이 안전합니다.

## Q29. 장애가 발생하면 전체 서비스가 멈추나요?

서비스가 STT, OCR, Parsed, Core로 분리되어 있어 특정 기능 장애가 전체 시스템 장애로 바로 이어지지 않도록 설계할 수 있습니다.

| 장애 지점 | 영향 | 대응 |
| --- | --- | --- |
| STT 장애 | 회의 음성 변환 불가 | 재시도, job 실패 상태 반환 |
| OCR 장애 | 문서 텍스트 추출 불가 | 문서 ingest 실패 처리 |
| Qdrant 장애 | RAG 검색 불가 | 근거 없음 응답 또는 fallback |
| LLM 장애 | 생성 기능 불가 | job 실패, 재시도 |

단, 운영 수준에서는 health check, retry, circuit breaker, 모니터링이 추가되어야 합니다.

## Q30. 이 구조의 가장 큰 리스크는 무엇인가요?

가장 큰 리스크는 AI 모델 자체보다 각 단계의 오류가 downstream으로 전파된다는 점입니다.

| 리스크 | 예시 | 대응 |
| --- | --- | --- |
| STT 오류 | 잘못 인식된 발화가 회의록에 반영 | transcript 검수/수정 UI |
| OCR 오류 | 표나 숫자 인식 오류 | OCR confidence, 후처리 |
| Retrieval 오류 | 관련 없는 문서 검색 | reranker, 평가셋 튜닝 |
| LLM 환각 | 근거 없는 결정사항 생성 | no-evidence policy, citation |
| 권한 오류 | 다른 프로젝트 문서 검색 | 백엔드 인가와 collection filter |

## 3. 발표장에서 바로 쓸 수 있는 답변

### 모델 선정 기준은 무엇인가요?

한국어 업무 데이터 적합성, 사내 배포 가능성, 모델 교체 가능성, 근거 기반 답변 품질을 기준으로 선택했습니다. 음성은 WhisperX, 문서는 PaddleOCR-VL과 MinerU, 검색은 한국어 임베딩과 Qdrant, 생성은 vLLM 기반 LLM으로 역할을 분리했습니다.

### 왜 Private RAG인가요?

회의록과 내부 문서는 사내 민감 정보이기 때문에 외부 API에 전적으로 의존하기보다, 문서와 벡터 DB를 통제 가능한 인프라에 두는 Private RAG 구조가 필요했습니다. 이를 통해 데이터 위치, 접근 권한, 로그, 보관 정책을 사내 기준에 맞출 수 있습니다.

### hallucination은 어떻게 줄이나요?

LLM이 단독으로 답변하지 않고 Qdrant에서 관련 문서와 이전 회의록을 검색한 뒤, 그 근거를 context로 제공해 답변하게 합니다. 또한 Parent-Child chunk로 검색 정확도와 문맥 보존을 함께 확보하고, 근거가 없으면 무리하게 답하지 않는 정책이 필요합니다.

### 가장 보완해야 할 점은 무엇인가요?

정량 평가셋 구축입니다. STT는 WER/CER, 검색은 recall@k/MRR, RAG 답변은 근거 일치율과 hallucination rate, 회의록은 결정사항과 todo 추출 정확도로 평가해야 합니다.
