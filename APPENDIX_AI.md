# Appendix. AI 모델 구조 Q&A

본 부록은 `ai/` 폴더의 구현을 바탕으로, 발표와 질의응답에서 설명하기 쉬운 **모델 중심 구조**로 정리한 문서입니다.  
코드 파일이나 실행 명령보다, 각 AI 모델이 어떤 문제를 해결하고 왜 필요한지를 중심으로 설명합니다.

---

## 1. 한눈에 보는 AI 모델 구성

이 프로젝트의 AI는 하나의 거대 모델이 모든 일을 처리하는 구조가 아니라, 작업 성격에 맞는 모델을 나누어 연결한 파이프라인 구조입니다.

| 단계 | 사용 모델/기술 | 담당 문제 |
| --- | --- | --- |
| 음성 인식 | WhisperX | 회의 녹음 파일을 발화자/시간 정보가 포함된 텍스트로 변환 |
| 문서 인식 | PaddleOCR-VL, Qwen VL | PDF/이미지 문서에서 텍스트와 표/시각 정보를 추출 |
| 문서 파싱 | MinerU | PDF 문서의 제목, 본문, 표, 페이지 구조를 분석 |
| 임베딩 | 한국어 Sentence Embedding 모델 | 문서와 회의록을 벡터로 변환해 검색 가능하게 저장 |
| 벡터 검색 | Qdrant | 프로젝트별 문서/회의록에서 관련 근거 검색 |
| 생성 모델 | vLLM 기반 Instruction LLM | 회의록, 안건, 준비자료, 챗봇 답변 생성 |

## 2. 모델 선정 기준 요약

모델은 단순히 최신 모델이나 가장 큰 모델을 기준으로 고르지 않았습니다. 발표에서 강조할 핵심은 **업무 흐름에 맞는 역할 분리와 운영 가능성**입니다.

| 기준 | 설명 |
| --- | --- |
| 한국어 적합성 | 회의록, 내부 문서, 사용자 질문이 대부분 한국어이므로 한국어 처리 품질을 우선 고려 |
| 업무 데이터 적합성 | 회의 음성, PDF, 표, 정책 문서, 이전 회의록처럼 실제 입력 형태에 맞는 모델 선택 |
| 운영 가능성 | RunPod/GPU 환경에서 실행 가능하고, API 서버로 분리 운영 가능한지 고려 |
| 교체 가능성 | 특정 모델 성능이 부족할 때 STT/OCR/LLM/Embedding을 독립적으로 바꿀 수 있는 구조 |
| 비용 효율 | 모든 작업을 대형 LLM 하나에 맡기지 않고, 전용 모델과 RAG로 처리 비용을 분산 |
| 근거 기반 답변 | LLM의 일반 지식보다 프로젝트 내부 문서와 회의록을 우선 참고하도록 설계 |

전체 AI 흐름은 다음과 같습니다.

```text
회의 음성 / 문서 업로드
        ↓
STT 또는 OCR로 원천 텍스트 추출
        ↓
문서 구조 분석 및 chunk 생성
        ↓
임베딩 모델로 벡터화
        ↓
Qdrant에 프로젝트별 저장
        ↓
LLM이 검색된 근거를 참고해 회의록, 안건, 준비자료, 챗봇 답변 생성
```

## Q1. 이 프로젝트에서 AI가 해결하는 핵심 문제는 무엇인가요?

AI가 해결하는 핵심 문제는 회의와 문서에서 발생하는 비정형 정보를 업무에 바로 사용할 수 있는 구조화된 정보로 바꾸는 것입니다.

사용자는 회의 녹음, PDF, 이미지, 내부 문서를 업로드합니다. 이 데이터는 그대로는 검색하거나 요약하기 어렵습니다. AI 파이프라인은 이를 텍스트로 변환하고, 의미 단위로 나누고, 벡터 DB에 저장한 뒤, 사용자의 질문이나 업무 목적에 맞는 답변을 생성합니다.

즉, 단순 챗봇이 아니라 다음 과정을 자동화합니다.

| 입력 | AI 처리 | 결과 |
| --- | --- | --- |
| 회의 녹음 | STT, 발화 구간 분석 | 회의 transcript |
| 회의 transcript | LLM 요약 및 구조화 | 회의록, todo, 결정사항 |
| PDF/이미지 | OCR, 문서 구조 분석 | 검색 가능한 문서 텍스트 |
| 내부 문서/회의록 | 임베딩, 벡터 검색 | 프로젝트별 지식 베이스 |
| 사용자 질문 | RAG + LLM | 근거 기반 답변 |

## Q2. 왜 하나의 모델이 아니라 여러 모델로 나누었나요?

회의 음성 인식, 문서 OCR, 문서 검색, 답변 생성은 서로 다른 종류의 AI 문제입니다.

하나의 LLM에 모든 작업을 맡기면 구현은 단순해 보일 수 있지만, 정확도와 운영 효율이 떨어집니다. 예를 들어 음성 인식은 STT 전용 모델이 더 정확하고, 문서 검색은 임베딩 모델과 벡터 DB를 사용하는 것이 더 안정적입니다.

따라서 역할을 다음처럼 분리했습니다.

| 분리 이유 | 설명 |
| --- | --- |
| 정확도 | 음성, 이미지, 검색, 생성에 특화된 모델을 각각 사용 |
| 재사용성 | STT, OCR, RAG, 생성 기능을 독립적으로 개선 가능 |
| 확장성 | 특정 모델만 교체하거나 성능 개선 가능 |
| 비용 효율 | 모든 요청을 대형 LLM에 보내지 않고 필요한 모델만 사용 |
| 장애 격리 | OCR 장애가 전체 챗봇 기능에 바로 영향을 주지 않도록 분리 |

## Q3. STT 모델은 어떤 역할을 하나요?

STT 모델은 회의 녹음 파일을 텍스트 transcript로 변환합니다.

이 프로젝트에서는 WhisperX를 사용합니다. WhisperX는 Whisper 기반 음성 인식에 alignment와 diarization 기능을 결합할 수 있어 회의 데이터 처리에 적합합니다.

| 항목 | 설명 |
| --- | --- |
| 입력 | 회의 녹음 파일 |
| 출력 | 발화 텍스트, 시간 정보, segment 정보 |
| 사용 목적 | 회의록 생성의 원천 데이터 생성 |
| 장점 | 긴 음성 처리, 발화 구간 분리, 한국어 음성 인식 |
| 선택 기능 | alignment, diarization |

회의록 생성 모델은 음성을 직접 듣는 것이 아니라, STT가 만든 transcript를 기반으로 회의 내용을 요약하고 구조화합니다. 따라서 STT 품질은 전체 회의록 품질에 직접적인 영향을 줍니다.

## Q4. OCR 모델은 어떤 역할을 하나요?

OCR 모델은 PDF나 이미지 문서에서 텍스트를 추출합니다.

회의 준비자료, 회의 첨부파일, 내부 정책 문서, 이미지 기반 문서는 일반 텍스트 검색이 어렵습니다. OCR 모델은 이런 문서를 텍스트로 변환해 RAG 검색에 사용할 수 있게 합니다.

| 항목 | 설명 |
| --- | --- |
| 기본 OCR 백엔드 | PaddleOCR-VL |
| 대체 경로 | Qwen VL |
| 입력 | PDF, 이미지 |
| 출력 | 문서에서 추출된 텍스트 |
| 목적 | 문서를 검색 가능한 지식으로 변환 |

PaddleOCR-VL은 문서 이미지, 표, 레이아웃이 포함된 자료를 처리하는 데 적합합니다. Qwen VL은 시각 정보를 이해하는 멀티모달 모델 경로로 사용할 수 있습니다.

## Q5. 문서 파싱 모델은 왜 필요한가요?

OCR이 텍스트를 추출하더라도, 문서를 단순한 긴 문자열로만 다루면 검색 품질이 떨어집니다.

예를 들어 규정 문서나 회의자료에는 제목, 장, 절, 표, 페이지 번호, 조항 번호 같은 구조가 있습니다. 이런 구조를 무시하고 일정 길이로만 자르면 답변의 근거가 끊기거나, 관련 없는 문맥이 섞일 수 있습니다.

이 프로젝트에서는 MinerU 기반 파싱을 통해 문서 구조를 분석합니다.

| 역할 | 설명 |
| --- | --- |
| 문서 구조 인식 | 제목, 본문, 표, 페이지 단위 분석 |
| 검색 단위 생성 | 의미가 끊기지 않는 chunk 생성 |
| 메타데이터 유지 | 문서명, 페이지, 섹션 등 근거 정보 보존 |
| RAG 품질 개선 | 답변 생성 시 더 정확한 문맥 제공 |

---

## Q6. 임베딩 모델은 어떤 역할을 하나요?

임베딩 모델은 텍스트를 숫자 벡터로 변환합니다.

LLM이 모든 문서를 매번 처음부터 읽는 방식은 비효율적입니다. 대신 문서와 회의록을 미리 벡터화해 Qdrant에 저장하고, 질문이 들어오면 의미적으로 가까운 chunk를 검색합니다.

| 항목 | 설명 |
| --- | --- |
| 입력 | 문서 chunk, 회의록 chunk, 사용자 질문 |
| 출력 | 의미를 표현하는 벡터 |
| 사용 목적 | 의미 기반 검색 |
| 기본 모델 | `jhgan/ko-sroberta-multitask` |

한국어 회의록과 내부 문서 검색이 핵심이기 때문에, 한국어 문장 의미 표현에 적합한 임베딩 모델을 사용합니다.

## Q7. RAG 구조는 왜 사용했나요?

RAG는 Retrieval-Augmented Generation의 약자로, LLM이 답변을 생성하기 전에 관련 문서나 회의록을 먼저 검색해 참고하도록 하는 구조입니다.

LLM만 사용하면 그럴듯하지만 근거가 부족한 답변이 나올 수 있습니다. RAG를 사용하면 프로젝트 내부 문서와 이전 회의록을 근거로 답변할 수 있습니다.

| LLM 단독 사용 | RAG 사용 |
| --- | --- |
| 모델이 알고 있는 일반 지식에 의존 | 프로젝트 내부 문서와 회의록을 검색 |
| 최신 내부 정보 반영 어려움 | 업로드된 문서와 회의 내용을 반영 |
| 환각 가능성 증가 | 검색 근거를 기반으로 답변 |
| 프로젝트별 맥락 구분 어려움 | 프로젝트별 collection으로 분리 가능 |

이 프로젝트에서 RAG는 회의 챗봇, 회의 준비자료 생성, 이전 회의 기반 답변에 사용됩니다.

## Q8. Parent-Child chunk 구조는 무엇인가요?

문서를 검색할 때는 작은 단위가 유리하지만, 답변을 만들 때는 넓은 문맥이 필요합니다.

이 문제를 해결하기 위해 Parent-Child chunk 구조를 사용합니다.

| 구분 | 역할 |
| --- | --- |
| Child chunk | 실제 벡터 검색에 사용하는 작은 텍스트 단위 |
| Parent context | child가 속한 문서, 페이지, 제목, 조항, 표 등 상위 문맥 |

예를 들어 "제3장 접근권한 관리" 안에 "제12조 권한 부여"가 있다면, child는 검색을 위한 본문 일부이고 parent는 문서명, 장, 조항, 페이지 정보를 포함합니다.

이 구조의 장점은 다음과 같습니다.

| 장점 | 설명 |
| --- | --- |
| 검색 정확도 | 작은 child 단위로 관련 문장을 잘 찾음 |
| 문맥 보존 | 답변 생성 시 parent 정보를 함께 제공 |
| 근거 추적 | 문서명, 페이지, 조항 정보를 유지 |
| 회의자료 적합성 | 표, 섹션, 페이지가 많은 문서에서 유리 |

정리하면 child는 "잘 찾기 위한 단위"이고, parent는 "제대로 이해하기 위한 문맥"입니다.

## Q9. 생성 모델은 어떤 역할을 하나요?

생성 모델은 검색된 문맥과 사용자 요청을 바탕으로 최종 답변을 생성합니다.

이 프로젝트에서는 vLLM 기반의 OpenAI-compatible serving 구조를 사용합니다. README 기준 예시 모델은 `cyankiwi/gemma-4-12B-it-AWQ-INT4`이며, 환경에 따라 llama.cpp 기반 GGUF 모델도 사용할 수 있습니다.

| 생성 작업 | 설명 |
| --- | --- |
| 회의록 생성 | STT transcript를 요약해 회의록, 결정사항, todo로 구조화 |
| 안건 생성 | 회의 목적과 프로젝트 맥락을 바탕으로 논의 안건 생성 |
| 준비자료 생성 | 이전 회의, 내부 문서, 외부 뉴스 기반 자료 생성 |
| 챗봇 답변 | Qdrant 검색 결과를 참고해 프로젝트 질의에 답변 |

생성 모델은 단독으로 답변하지 않고, 가능한 경우 RAG 검색 결과를 함께 참고합니다. 이를 통해 프로젝트 내부 맥락과 근거를 반영할 수 있습니다.

## Q10. 회의록 생성은 어떤 방식으로 이루어지나요?

회의록 생성은 STT 결과를 LLM이 구조화하는 방식입니다.

전체 흐름은 다음과 같습니다.

```text
회의 녹음
  ↓
WhisperX STT
  ↓
발화 시간, speaker, transcript 생성
  ↓
LLM이 회의 목적에 맞게 요약
  ↓
회의록, 결정사항, 액션 아이템 생성
  ↓
생성된 회의록을 Qdrant에 적재해 이후 검색에 활용
```

회의록 생성에서 중요한 점은 단순 요약이 아니라 업무에 필요한 구조로 변환하는 것입니다.

| 출력 항목 | 의미 |
| --- | --- |
| 회의 요약 | 전체 논의의 핵심 요약 |
| 주요 논의 내용 | 안건별 상세 내용 |
| 결정사항 | 회의에서 확정된 내용 |
| Todo | 담당자, 할 일, 기한 중심 정리 |
| 후속 검색 활용 | 생성된 회의록을 다시 RAG 지식으로 저장 |

## Q11. 회의 준비자료 생성은 어떻게 이루어지나요?

회의 준비자료는 단순히 LLM이 새로 작성하는 것이 아니라, 여러 정보원을 조합해 생성합니다.

| 정보원 | 활용 방식 |
| --- | --- |
| 이전 회의록 | 지난 논의, 결정사항, 미완료 todo 확인 |
| 내부 문서 | 프로젝트 정책, 요구사항, 자료 근거 검색 |
| 외부 뉴스 | 관련 최신 이슈나 산업 동향 참고 |
| 사용자 입력 | 회의 목적, 참석자, 주제 반영 |

이 구조는 회의 전에 필요한 맥락을 자동으로 모아주는 역할을 합니다. 특히 프로젝트별 Qdrant collection을 사용하므로, 다른 프로젝트의 문서가 섞이지 않도록 설계되어 있습니다.

## Q12. 모델 환각은 어떻게 줄이나요?

LLM은 근거 없이 그럴듯한 답변을 만들 수 있습니다. 이를 줄이기 위해 이 프로젝트는 RAG와 메타데이터 기반 문맥 제공 방식을 사용합니다.

| 방법 | 효과 |
| --- | --- |
| Qdrant 검색 | 답변 전에 관련 문서/회의록을 먼저 찾음 |
| Parent-Child chunk | 검색 정확도와 문맥 보존을 함께 확보 |
| 프로젝트별 collection | 다른 프로젝트 문서가 섞이는 문제 감소 |
| 프롬프트 분리 | 회의록, 안건, 준비자료, 챗봇 목적별 지시 분리 |
| 근거 부족 처리 | 관련 문서가 없을 때 무리한 답변을 줄임 |

즉, LLM의 창의성보다 프로젝트 문서 기반의 근거성을 우선하도록 설계했습니다.

## Q13. 모델 선택 기준은 무엇인가요?

모델 선택 기준은 단순히 성능이 높은 모델을 고르는 것이 아니라, 프로젝트 요구사항과 운영 환경에 맞는지를 기준으로 했습니다.

| 기준 | 설명 |
| --- | --- |
| 한국어 처리 | 회의록, 내부 문서, 질문이 한국어 중심 |
| 긴 입력 처리 | 회의 transcript와 문서 context를 다룰 수 있어야 함 |
| GPU 운영 가능성 | RunPod/GPU 환경에서 실행 가능해야 함 |
| 교체 가능성 | 모델을 바꿔도 API 구조를 유지할 수 있어야 함 |
| 비용 효율 | 모든 기능을 대형 모델 하나에 의존하지 않도록 분리 |
| RAG 적합성 | 검색 결과를 바탕으로 안정적으로 답변 가능해야 함 |

## Q14. 성능은 어떻게 검증할 수 있나요?

발표나 심사에서 성능 검증 질문이 나오면, 모델별로 평가 기준을 나누어 설명하는 것이 좋습니다.

| 영역 | 검증 기준 |
| --- | --- |
| STT | 발화 누락 여부, 한국어 인식 정확도, 긴 회의 처리 안정성 |
| OCR | 문서 텍스트 추출 정확도, 표/페이지 구조 보존 여부 |
| 임베딩/RAG | 질문과 관련된 문서를 상위 결과로 찾는지 |
| 회의록 생성 | 결정사항, todo, 담당자, 기한을 잘 구조화하는지 |
| 챗봇 답변 | 내부 문서 근거와 답변이 일치하는지 |
| 운영 안정성 | 긴 작업에서 timeout 없이 job 처리되는지 |

정량 평가를 한다면 STT는 WER, 검색은 top-k hit rate, RAG 답변은 근거 일치율과 사용자 평가를 사용할 수 있습니다.

## Q17. WhisperX를 STT 모델로 선택한 이유는 무엇인가요?

WhisperX는 단순 음성 인식뿐 아니라 회의 데이터에 필요한 alignment와 diarization 확장이 가능하다는 점에서 적합합니다.

회의록 자동화에서는 텍스트만 얻는 것보다 "언제 누가 어떤 말을 했는지"가 중요합니다. WhisperX는 Whisper 기반 인식 품질을 활용하면서 발화 구간 정렬과 화자 분리 기능을 함께 고려할 수 있습니다.

| 비교 모델 | 장점 | 한계 |
| --- | --- | --- |
| Whisper | 범용 STT 품질 우수, 사용 사례 많음 | alignment/diarization은 별도 구성 필요 |
| faster-whisper | 추론 속도와 메모리 효율 우수 | 회의 구조화 기능은 별도 구현 필요 |
| WhisperX | Whisper 기반 + alignment/diarization 확장 가능 | diarization은 추가 설정과 HF token이 필요할 수 있음 |
| Google Speech-to-Text | 운영형 API로 안정적 | 비용, 외부 전송, 커스터마이징 제약 |
| Naver Clova Speech | 한국어 서비스 친화적 | 외부 API 의존성과 비용 고려 필요 |

선정 이유를 한 문장으로 말하면, WhisperX는 회의록 생성에 필요한 "음성 인식 + 발화 구간 정보"를 함께 다루기 좋기 때문입니다.

## Q18. PaddleOCR-VL을 OCR 기본 백엔드로 선택한 이유는 무엇인가요?

PaddleOCR-VL은 일반 텍스트 OCR뿐 아니라 문서 이미지, 표, 레이아웃이 포함된 자료를 처리하는 데 적합합니다.

이 프로젝트의 문서 입력은 단순 이미지보다 PDF, 회의자료, 표가 포함된 내부 문서일 가능성이 높습니다. 따라서 글자만 읽는 OCR보다 문서형 자료에 강한 OCR 백엔드가 필요했습니다.

| 비교 모델/기술 | 장점 | 한계 |
| --- | --- | --- |
| Tesseract OCR | 가볍고 오픈소스, 설치 쉬움 | 복잡한 문서 레이아웃과 한국어 문서에서 품질 한계 |
| EasyOCR | 사용이 간단하고 다국어 지원 | 긴 PDF나 표 구조 처리에는 제한 |
| Google Vision OCR | 높은 인식률과 안정적인 API | 비용, 외부 전송, API 의존성 |
| Qwen VL | 이미지 이해와 추론 가능 | OCR 전용 파이프라인보다 비용과 추론 부담 큼 |
| PaddleOCR-VL | 문서 이미지와 레이아웃 처리에 강점 | serving 프로세스와 GPU 자원 관리 필요 |

## Q19. Qwen VL은 왜 OCR 대체 경로로 두었나요?

Qwen VL은 단순 OCR보다 시각 정보를 이해하는 멀티모달 모델에 가깝습니다.

따라서 문서 이미지에서 글자만 추출하는 것이 아니라, 표나 이미지 맥락을 설명해야 하는 경우 보조 경로로 사용할 수 있습니다. 다만 모든 OCR 요청을 Qwen VL로 처리하면 비용과 추론 시간이 커질 수 있어 기본 경로는 PaddleOCR-VL로 두고, 필요할 때 대체 경로로 사용하는 구성이 더 현실적입니다.

| 구분 | PaddleOCR-VL | Qwen VL |
| --- | --- | --- |
| 강점 | OCR/문서 추출에 특화 | 이미지 맥락 이해 가능 |
| 적합한 작업 | PDF, 문서, 표 텍스트 추출 | 복잡한 이미지 설명, 시각 질의응답 |
| 운영 비용 | 상대적으로 OCR 파이프라인에 적합 | 요청당 추론 부담이 큼 |
| 프로젝트 내 위치 | 기본 OCR 백엔드 | 보조 또는 대체 OCR 경로 |

## Q22. Qdrant를 벡터 DB로 선택한 이유는 무엇인가요?

Qdrant는 벡터 검색을 위한 전용 데이터베이스이며, payload metadata와 collection 분리 관리가 쉽습니다.

이 프로젝트는 프로젝트별 문서와 회의록을 분리해야 하므로, collection 단위 관리와 metadata filter가 중요합니다.

| 비교 기술 | 장점 | 한계 |
| --- | --- | --- |

| Elasticsearch | 키워드 검색과 운영 경험 풍부 | 벡터 검색 중심 설계는 별도 튜닝 필요 |
| Pinecone | 관리형 벡터 DB로 운영 편리 | 외부 서비스 비용과 의존성 |
| Qdrant | 벡터 검색, metadata, collection 관리에 적합 | 별도 서버 운영 필요 |

프로젝트별 collection 이름을 `hpm_project_<project_id>` 형태로 관리할 수 있어 멀티 프로젝트 구조에 잘 맞습니다.

## Q23. vLLM을 생성 모델 serving에 사용한 이유는 무엇인가요?

vLLM은 LLM을 OpenAI-compatible API 형태로 serving하기 좋고, GPU 환경에서 추론 효율을 높일 수 있습니다.

생성 모델은 회의록, 안건, 준비자료, 챗봇 답변처럼 요청 길이가 길고 응답도 긴 작업을 처리합니다. 따라서 단순 로컬 실행보다 API serving 구조가 필요합니다.

| 비교 방식 | 장점 | 한계 |
| --- | --- | --- |
| Hugging Face Transformers 직접 실행 | 자유도가 높고 구현이 직접적 | 서버화, 동시성, 메모리 관리 부담 |
| llama.cpp | GGUF 모델을 비교적 가볍게 운영 가능 | 고성능 GPU serving에는 한계가 있을 수 있음 |
| Ollama | 로컬 테스트와 모델 관리가 쉬움 | 운영형 API/대규모 동시 처리 관점에서는 추가 검토 필요 |
| OpenAI API | 품질과 안정성이 높음 | 비용, 외부 전송, 모델 통제 제약 |
| vLLM | GPU serving, OpenAI-compatible API, 운영 확장성 | GPU 환경 구성 필요 |

## Q24. AWQ/INT4 계열 모델을 고려한 이유는 무엇인가요?

README 기준 예시 생성 모델은 `cyankiwi/gemma-4-12B-it-AWQ-INT4`입니다.

AWQ/INT4 모델은 원본 대형 모델보다 메모리 사용량을 줄여 GPU 환경에서 운영하기 쉽게 만든 양자화 모델입니다. 회의록 생성과 RAG 챗봇은 긴 입력을 처리해야 하므로, 모델 성능뿐 아니라 실제 GPU에 올릴 수 있는지가 중요합니다.

| 구분 | 장점 | 고려사항 |
| --- | --- | --- |
| FP16 대형 모델 | 품질이 높을 가능성 | GPU 메모리 요구량 큼 |
| INT4/AWQ 모델 | 메모리 효율과 serving 가능성 향상 | 일부 품질 저하 가능성 |
| API 기반 대형 모델 | 품질과 안정성 우수 | 비용과 외부 전송 이슈 |
| 소형 로컬 모델 | 비용과 속도 측면 유리 | 긴 회의록/문서 기반 답변 품질 한계 가능 |

선정 이유는 제한된 GPU 자원에서 회의록 생성과 RAG 답변을 현실적으로 운영하기 위해서입니다.

## Q25. 왜 Private RAG 구조를 고려했나요?

이 프로젝트는 사내에 배포될 가능성을 전제로 하기 때문에, 내부 회의록과 문서를 외부 서비스에 그대로 의존해 처리하기보다 Private RAG 구조가 더 적합합니다.

회의 녹음, 회의록, 내부 정책 문서, 프로젝트 자료에는 민감한 업무 정보가 포함될 수 있습니다. Private RAG는 이런 데이터를 사내 또는 통제 가능한 인프라 안에서 벡터화하고 검색하며, LLM은 검색된 내부 근거를 바탕으로 답변하도록 만드는 구조입니다.

| 기준 | 외부 API 중심 구조 | Private RAG 구조 |
| --- | --- | --- |
| 데이터 위치 | 요청 시 외부 API로 데이터 전송 가능성 존재 | 내부망 또는 통제 가능한 서버에 문서와 벡터 저장 |
| 보안/컴플라이언스 | 외부 서비스 정책과 계약 조건 확인 필요 | 사내 보안 정책에 맞춰 접근 제어 가능 |
| 권한 관리 | 서비스 외부에서 세밀한 프로젝트 권한 반영이 어려울 수 있음 | 프로젝트별 collection과 사용자 권한을 연계 가능 |
| 감사 가능성 | 어떤 내부 근거로 답했는지 추적이 제한될 수 있음 | 문서명, 페이지, 회의 ID 등 근거 추적 가능 |
| 운영 통제 | 모델, 저장 위치, 로그 정책을 외부 정책에 의존 | 모델, DB, 로그, 보관 정책을 직접 통제 가능 |
| 비용 구조 | 사용량 증가에 따라 API 비용 증가 | 인프라 비용 중심으로 예측 가능 |

따라서 이 프로젝트의 RAG 구조는 단순히 LLM 비용을 줄이기 위한 선택이 아니라, 사내 배포 환경에서 내부 문서와 회의 데이터를 안전하게 활용하기 위한 설계입니다.

발표에서는 다음처럼 답하면 좋습니다.

> 이 시스템은 회의록과 내부 문서를 다루기 때문에 사내 배포를 고려했습니다. 그래서 문서와 벡터 DB를 내부 인프라에 두고, 프로젝트별로 분리된 Private RAG 구조를 설계했습니다. 이를 통해 데이터 외부 반출 위험을 줄이고, 권한 관리와 근거 추적이 가능한 형태로 답변을 생성할 수 있습니다.

## Q26. 왜 RAG를 쓰고 fine-tuning을 먼저 하지 않았나요?

프로젝트 내부 문서와 회의록은 계속 바뀝니다.

fine-tuning은 모델의 말투나 특정 작업 형식을 학습시키는 데 유리하지만, 자주 바뀌는 내부 지식을 반영하려면 매번 재학습이 필요합니다. 반면 RAG는 문서를 새로 업로드하고 벡터 DB에 적재하면 최신 내용을 바로 검색에 반영할 수 있습니다.

| 구분 | RAG | Fine-tuning |
| --- | --- | --- |
| 최신 문서 반영 | 빠름 | 재학습 필요 |
| 내부 근거 추적 | 쉬움 | 모델 내부 지식이라 추적 어려움 |
| 비용 | 문서 적재와 검색 비용 중심 | 학습 비용 발생 |
| 적합한 목적 | 지식 검색, 근거 기반 답변 | 말투, 형식, 특정 패턴 학습 |

이 프로젝트는 내부 문서와 회의록이 계속 쌓이는 구조이므로 RAG가 우선순위에 더 적합합니다.

## Q27. Reranker는 왜 선택 기능으로 두었나요?

Reranker는 벡터 검색 결과를 다시 정렬해 더 관련성 높은 문서를 위로 올리는 역할을 합니다.

다만 모든 요청에 reranker를 적용하면 latency와 비용이 증가할 수 있습니다. 그래서 기본 검색은 Qdrant top-k로 처리하고, 더 높은 정확도가 필요한 경우 Cohere 같은 reranker를 선택적으로 사용할 수 있게 하는 구조가 적합합니다.

| 방식 | 장점 | 한계 |
| --- | --- | --- |
| Vector search only | 빠르고 단순 | 의미가 비슷하지만 덜 중요한 문서가 상위에 올 수 있음 |
| Reranker 추가 | 최종 context 품질 향상 | 비용과 응답 시간 증가 |
| Hybrid search | 키워드와 벡터 검색 결합 가능 | 인덱스와 튜닝 복잡도 증가 |

## Q28. 외부 뉴스 검색은 왜 추가했나요?

회의 준비자료는 내부 문서만으로 충분하지 않은 경우가 있습니다.

예를 들어 프로젝트 주제가 정책, 시장, 기술 트렌드와 관련되어 있다면 최신 외부 정보가 필요할 수 있습니다. 이때 Naver News API를 활용하면 회의 준비자료에 외부 동향을 보조 정보로 포함할 수 있습니다.

| 정보원 | 역할 |
| --- | --- |
| 이전 회의록 | 지난 결정사항과 미완료 작업 확인 |
| 내부 문서 | 프로젝트 내부 근거 확인 |
| 외부 뉴스 | 최신 이슈와 시장/기술 동향 보완 |

단, 외부 뉴스는 내부 근거가 아니라 참고 정보이므로 내부 문서 기반 RAG와 구분해 사용하는 것이 중요합니다.

## Q29. 모델 교체가 쉬운 구조인가요?

네. 이 구조는 모델을 직접 코드에 강하게 고정하기보다 서비스와 환경변수 기준으로 분리합니다.

예를 들어 STT 모델을 바꾸더라도 회의록 생성 API 전체를 바꿀 필요는 없습니다. OCR 모델이나 임베딩 모델도 해당 서비스 내부에서 교체할 수 있습니다.

| 교체 대상 | 교체 예시 |
| --- | --- |
| STT | WhisperX → faster-whisper 또는 외부 STT API |
| OCR | PaddleOCR-VL → Qwen VL 또는 Google Vision OCR |
| Embedding | Ko-SRoBERTa → BGE-M3 또는 OpenAI embedding |
| Vector DB | Qdrant → Pinecone, Chroma, Elasticsearch |
| LLM | vLLM serving 모델 → llama.cpp 모델 또는 외부 OpenAI-compatible API |

이 점은 발표에서 "모델 종속적인 구현"이 아니라 "모델 교체 가능한 파이프라인"이라고 설명할 수 있습니다.

## Q30. 심사위원이 모델 선정 근거를 물으면 어떻게 답하면 되나요?

다음처럼 답하면 좋습니다.

이 프로젝트는 회의 자동화와 내부 문서 기반 질의응답이 핵심이기 때문에, 하나의 LLM보다 작업별 전문 모델을 조합하는 방식을 선택했습니다. 음성은 WhisperX, 문서는 PaddleOCR-VL과 MinerU, 검색은 한국어 임베딩 모델과 Qdrant, 최종 생성은 vLLM 기반 LLM이 담당합니다. 이렇게 나누면 각 단계의 정확도를 개별적으로 개선할 수 있고, 모델 교체도 쉽습니다. 또한 RAG를 사용해 LLM이 내부 문서와 이전 회의록을 근거로 답변하도록 하여 환각을 줄였습니다.

짧게 말하면, 선정 기준은 **한국어 업무 데이터 적합성, 운영 가능성, 교체 가능성, 근거 기반 답변 품질**입니다.

## 발표용 요약 답변

### AI 구조를 한 문장으로 설명하면?

회의 음성과 내부 문서를 STT, OCR, 임베딩 모델로 지식화하고, LLM이 RAG 검색 결과를 근거로 회의록, 안건, 준비자료, 챗봇 답변을 생성하는 구조입니다.

### 가장 중요한 모델 설계 포인트는?

하나의 LLM에 모든 작업을 맡기지 않고, 음성 인식, 문서 인식, 의미 검색, 답변 생성을 역할별 모델로 분리했다는 점입니다.

### RAG를 쓴 이유는?

LLM이 일반 지식만으로 답변하지 않고, 프로젝트 내부 문서와 이전 회의록을 검색한 뒤 근거 기반으로 답변하게 하기 위해서입니다.

### Parent-Child chunk를 쓴 이유는?

작은 chunk로 검색 정확도를 높이면서도, parent 문맥을 함께 제공해 문서의 제목, 페이지, 조항, 표 정보가 끊기지 않게 하기 위해서입니다.

### 모델 구조의 차별점은?

회의록 생성에 그치지 않고, 생성된 회의록과 업로드 문서를 다시 Qdrant에 저장해 이후 회의 준비자료와 챗봇 답변에 재사용하는 순환형 지식 구조입니다.
